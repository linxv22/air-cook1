# 智能空气炸锅 云服务器接口文档

> **适用对象**：Python 服务器开发者  
> **服务器 IP**：`8.162.21.140`  
> **WebSocket 端口**：`8765`  
> **说明**：服务器负责中转 APP 与 ESP32 之间的所有通信。

---

## 1. 整体架构

```
┌─────────────┐     HTTP (用户直接输入服务器地址连接)        ┌──────────────┐     WebSocket (ws://8.162.21.140:8765)     ┌─────────────┐
│  手机 APP   │ ←──────────────────────────────────────────→ │  Python 服务器 │ ←───────────────────────────────────→ │ ESP32-S3    │
│  (React     │                                              │  (FastAPI/    │                                       │  (空气炸锅)  │
│   Native)   │                                              │   aiohttp)    │                                       │             │
└─────────────┘                                              └──────────────┘                                       └─────────────┘
                                                                  │
                                                                  │ (数据库操作)
                                                                  ↓
                                                         ┌──────────────┐
                                                         │  Supabase    │
                                                         │  Database    │
                                                         └──────────────┘
```

| 通信链路 | 协议 | 说明 |
|----------|------|------|
| APP ↔ Supabase | HTTPS | 用户认证（注册/登录/会话管理） |
| APP ↔ Supabase DB | HTTPS | 查询食谱、历史记录、设备列表 |
| APP ↔ Supabase Edge Function | HTTPS | 调用 `fryer-api` 代理，转发请求到 Python 服务器 |
| Supabase Edge Function ↔ Python 服务器 | HTTP | 转发 API 请求 |
| Python 服务器 ↔ ESP32 | WebSocket | 长连接，双向通信，无需 `device_token` |
| ESP32 ↔ Python 服务器 | WebSocket | 运行中每 5 秒、空闲每 30 秒上报温度状态 |

## 2. APP 绑定方式与设备 ID 缓存机制

APP 登录后**不扫码**，改为用户**手动输入服务器 IP 地址和端口号**：

| 输入项 | 默认值 | 说明 |
|--------|--------|------|
| IP 地址 | `8.162.21.140` | 服务器公网 IP，用户可修改 |
| 端口号 | `8765` | WebSocket/HTTP 服务端口，用户可修改 |

用户点击"连接"后，APP 调用 `POST /api/bind`，服务器返回 `device_id`。**APP 将此 device_id 缓存在内存中**，后续所有操作（状态轮询、下发指令）直接使用这个 ID，**不再依赖 Supabase devices 表**。

> ⚠️ **关键设计**：APP 不直接操作 Supabase devices 表。设备绑定由服务器全权管理（内存 + 可选的 Supabase 写入），APP 只负责保存 device_id 并用它调用 API。这样即使 Supabase 不可用，APP 也能正常工作。

---

## 3. 在线判定（无心跳包）

服务器**不依赖心跳包（ping/pong）**来判定设备在线状态，而是通过实际数据包的接收时间来判断：

| 判定规则 | 实现方式 |
|----------|---------|
| 在线 | 距离**上一次收到**该 ESP32 的 `type: "status"` 数据包 **不超过 60 秒** |
| 离线 | 距离上一次收到 `type: "status"` 数据包**超过 60 秒** |

```python
# 在线判定伪代码
def is_device_online(connection):
    now = time.time()
    return (now - connection.last_data_received_at) < 60  # 秒
```

> 这要求 ESP32 在上报数据时严格遵守频率：运行中 ≤ 5 秒一次，空闲 ≤ 30 秒一次。服务器只需记录每次收到数据的时间戳，定时检查即可。

## 4. HTTP API 接口（APP 通过 Supabase Edge Function 调用）

> 所有接口路径前缀：`/api/`  
> Supabase Edge Function 名：`fryer-api`  
> Edge Function 作用：接收 APP 请求，转发到 Python 服务器 `http://8.162.21.140:8765`，将响应返回给 APP

### 4.1 设备绑定

```
POST /api/bind
```

**请求体**：
```json
{
  "user_id": "uuid-from-supabase-auth"
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_id` | string | Supabase Auth 的用户 UUID |

**成功响应**（200）：
```json
{
  "success": true,
  "message": "绑定成功",
  "device_id": "uuid-of-device-record"
}
```

**失败响应**（404）：
```json
{
  "success": false,
  "message": "没有可用的设备连接"
}
```

**服务器处理逻辑**：
1. 检查当前是否有 ESP32 设备已通过 WebSocket 连接
2. 如有可用设备，将 `user_id` 与当前 WebSocket 会话关联（存入 Supabase `devices` 表）
3. 通过 WebSocket 通知 ESP32 绑定成功
4. 如无可用设备连接，返回失败

> 📌 简化绑定流程：不再需要 `device_token`。服务器上连接了哪个 ESP32，用户就绑定哪个。一对一关系。

---

### 4.2 获取设备状态

```
GET /api/status/{device_id}
```

**URL 参数**：
| 参数 | 类型 | 说明 |
|------|------|------|
| `device_id` | string | 设备 UUID（绑定后获得，存储在 Supabase `devices` 表中） |

**成功响应**（200）：
```json
{
  "type": "status",
  "temp": 185.5,
  "target_temp": 200,
  "time_left": 415,
  "state": "running",
  "food_name": "鸡翅",
  "fan_speed": "High"
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `temp` | float | 当前炉内温度（摄氏度） |
| `target_temp` | integer | 目标温度，running 时存在，idle/stopped 时为 0 |
| `time_left` | integer | 剩余秒数，stopped 时为 0 |
| `state` | string | `"idle"` / `"running"` / `"stopped"` |
| `food_name` | string | 当前食物名称，idle 时为空 |
| `fan_speed` | string | `"High"` / `"Medium"` / `"Low"` |

**失败响应**（404）：
```json
{
  "success": false,
  "message": "设备离线"
}
```

**服务器处理逻辑**：
1. 从内存缓存中查找该设备的最新状态数据
2. 如果设备在 60 秒内有过数据上报，返回缓存数据
3. 如果超过 60 秒未收到数据，返回"设备离线"
4. 响应时间应 < 50ms（读内存缓存，不查数据库）

> 📌 APP 每 2 秒轮询一次此接口。

| `fan_speed` | string | 风扇转速（手动模式必需） |
| `recipe_id` | string | 食谱 UUID（食谱模式必需） |

**成功响应**（200）：

3. 检查设备在线状态（60 秒内有数据上报）
4. 通过 WebSocket 向 ESP32 下发 cook 指令
5. 等待 ESP32 确认（5 秒超时），返回对应结果

| `temp` | integer | 目标温度 40-220 |
| `time` | integer | 烹饪时长（秒） |
| `fan_speed` | string | 风扇转速 |
| `scheduled_at` | string | 预约时间，格式 `"YYYY-MM-DD-HH:mm:ss"`（必须晚于当前时间） |

**成功响应**（200）：
```json
{
  "success": true,
  "message": "预约成功",
  "schedule_id": "uuid-of-schedule-record"
}
```

**失败响应**：
```json
{
  "success": false,
  "message": "预约时间不能早于当前时间"
}
```

**服务器处理逻辑**：
1. 校验 `scheduled_at` > 当前时间
2. 将预约记录写入 Supabase `schedules` 表
3. 启动服务器端定时器（使用 `asyncio.call_later` 或 APScheduler）
4. 在 `scheduled_at` 前 **2 秒**通过 WebSocket 向 ESP32 下发 schedule 指令
5. 如果到达预约时间时设备离线，标记预约状态为 `"cancelled"`

## 5. WebSocket 服务端（Python 服务器 ↔ ESP32）

### 5.1 服务端职责

| 职责 | 说明 |
|------|------|
| 连接管理 | 维护 ESP32 设备的 WebSocket 长连接 |
| 在线判定 | 记录每次收到数据的时间戳，超过 60 秒无数据→离线 |
| 数据缓存 | 缓存每个设备的最新状态（内存中） |
| 指令下发 | 将 APP 请求转为 WebSocket 指令下发到 ESP32 |
| 状态返回 | 响应 APP 的 `/api/status/{device_id}` 查询 |

### 5.2 连接池与状态缓存

```python
# Python 伪代码示意
import time
from dataclasses import dataclass, field

@dataclass
class DeviceSession:
    websocket: any                          # WebSocket 连接对象
    user_id: str = None                     # 绑定的用户 ID
    status: dict = field(default_factory=lambda: {
        "temp": 0.0, "target_temp": 0,
        "time_left": 0, "state": "idle",
        "food_name": "", "fan_speed": "Low"
    })
    last_data_at: float = 0.0                # 上次收到数据的时间戳（用于在线判定）
    connected_at: float = 0.0                # 连接时间

# 连接池：WebSocket 会话 ID → DeviceSession
sessions: dict[str, DeviceSession] = {}
```

### 5.3 接收消息（ESP32 → 服务器）

ESP32 只需上报一种类型的消息：

```json
{
  "type": "status",
  "temp": 185.5,
  "target_temp": 200,
  "time_left": 415,
  "state": "running",
  "food_name": "鸡翅",
  "fan_speed": "High"
}
```

| 上报频率 | 状态 |
|----------|------|
| 每 5 秒 | `state = "running"` |
| 每 5 秒 | `state = "stopped"`（连续 2 次后切换为 idle） |
| 每 30 秒 | `state = "idle"` |

**服务器收到后的处理**：
```python
async def handle_status(session_id: str, data: dict):
    session = sessions[session_id]
    session.status = data                          # 更新内存缓存
    session.last_data_at = time.time()             # 刷新在线时间戳
    # 注意：不需要每次写数据库，APP 轮询时直接读内存缓存
```

### 5.4 下发消息（服务器 → ESP32）

| type | 触发条件 | 说明 |
|------|---------|------|

## 6. Supabase 数据库表结构

### 6.1 devices 表（设备注册）

```sql
CREATE TABLE IF NOT EXISTS devices (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  owner_id    UUID REFERENCES auth.users(id),
  name        VARCHAR(100) DEFAULT '我的空气炸锅',
  is_online   BOOLEAN DEFAULT false,
  created_at  TIMESTAMPTZ DEFAULT now()
);

-- ★ 重要：关闭 RLS，否则 APP 和服务器的 Supabase 读写会被拦截
ALTER TABLE devices DISABLE ROW LEVEL SECURITY;
```

> 去掉 `device_token` 字段。设备通过 WebSocket 连接自动注册，`id` 为主键。APP 绑定后缓存 `device_id` 到内存，不依赖此表即可工作，但建议保留此表用于服务端记录。

### 6.2 recipes 表（食谱库）

```sql
CREATE TABLE recipes (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name         VARCHAR(200) NOT NULL,
  category     VARCHAR(50) NOT NULL CHECK (category IN ('肉类', '海鲜', '素食', '烘焙')),
  temp         INTEGER NOT NULL CHECK (temp >= 40 AND temp <= 220),
  time_seconds INTEGER NOT NULL CHECK (time_seconds >= 60 AND time_seconds <= 3600),
  fan_speed    VARCHAR(10) NOT NULL CHECK (fan_speed IN ('Low', 'Medium', 'High')),
  tips         TEXT,
  image_url    TEXT,
  created_at   TIMESTAMPTZ DEFAULT now()
);
```

### 6.3 cook_history 表（烹饪历史）

```sql
CREATE TABLE cook_history (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  device_id    UUID REFERENCES devices(id),
  user_id      UUID REFERENCES auth.users(id),
  food_name    VARCHAR(200),
  temp         INTEGER,
  time_seconds INTEGER,
  fan_speed    VARCHAR(10),
  recipe_id    UUID REFERENCES recipes(id),
  started_at   TIMESTAMPTZ DEFAULT now(),
  finished_at  TIMESTAMPTZ
);
```

### 6.4 schedules 表（预约记录）

```sql
CREATE TABLE schedules (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  device_id    UUID REFERENCES devices(id),
  user_id      UUID REFERENCES auth.users(id),
  food_name    VARCHAR(200),
  temp         INTEGER,
  time_seconds INTEGER,
  fan_speed    VARCHAR(10),
  scheduled_at VARCHAR(20) NOT NULL,
  status       VARCHAR(20) DEFAULT 'pending' CHECK (status IN ('pending', 'sent', 'cancelled')),
  created_at   TIMESTAMPTZ DEFAULT now()
);
```

## 7. 服务器核心实现（Python 伪代码）

```python
import asyncio, json, time
from aiohttp import web

sessions = {}  # session_id → DeviceSession

async def ws_handler(request):
    ws = web.WebSocketResponse()
    await ws.prepare(request)
    session_id = str(id(ws))
    sessions[session_id] = DeviceSession(
        websocket=ws,
        connected_at=time.time(),
        last_data_at=time.time()
    )

    async for msg in ws:
        data = json.loads(msg.data)
        if data.get("type") == "status":
            sessions[session_id].status = data
            sessions[session_id].last_data_at = time.time()

    # 断开时清理
    del sessions[session_id]
    return ws

async def get_status(request):
    """GET /api/status/{device_id}"""
    # 找到对应用户的 ESP32 会话
    session = find_session_by_device_id(request.match_info['device_id'])
    if session is None:
        raise web.HTTPNotFound(text=json.dumps({"success": False, "message": "设备离线"}))

    # 检查在线状态（60 秒内有数据上报）
    if time.time() - session.last_data_at > 60:
        raise web.HTTPNotFound(text=json.dumps({"success": False, "message": "设备离线"}))

    return web.json_response(session.status)

async def start_cook(request):
    """POST /api/cook"""
    body = await request.json()
    session = find_session_by_device_id(body['device_id'])
    if session is None or time.time() - session.last_data_at > 60:
        raise web.HTTPBadRequest(text=json.dumps({"success": False, "message": "设备离线，下发失败"}))

    command = {"type": "cook", "food": body['food'], "temp": body['temp'],
               "time": body['time'], "fan_speed": body['fan_speed']}
    await session.websocket.send_json(command)
    # 等待 ACK（可选，5秒超时）
    try:
        ack = await asyncio.wait_for(session.websocket.receive(), timeout=5.0)
    except asyncio.TimeoutError:
        pass  # ESP32 可能不回复 ACK，指令已发送

    return web.json_response({"success": True, "message": "指令已下发"})

def is_device_online(session):
    """60 秒内收到过数据 → 在线"""
    return time.time() - session.last_data_at < 60
```

## 8. 定时任务

| 任务 | 频率 | 说明 |
|------|------|------|
| 在线状态刷新 | 每 30 秒 | 遍历所有会话，检查 `last_data_at`，更新 Supabase `devices.is_online` |
| 预约检查 | 每 10 秒 | 扫描 `schedules` 表中 `status=pending` 且 `scheduled_at` 已到达的记录 |
| 停止状态检测 | 每次收到 status | 检测到 `state: "stopped"` → 写入 `cook_history` 结束记录 |
