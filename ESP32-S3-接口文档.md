# 智能空气炸锅 ESP32-S3 接口文档

> **适用对象**：ESP32-S3 固件开发者  
> **通信架构**：ESP32-S3 ←→ Python 云服务器 ←→ 手机 APP  
> **说明**：ESP32-S3 不直接与 APP 通信，所有数据通过云服务器中转。

---

## 1. 设备连接方式

### 1.1 连接信息

| 项目 | 值 |
|------|-----|
| 服务器地址 | `ws://8.162.21.140:8765` |
| 连接时机 | ESP32 联网成功后立即建立并保持长连接 |
| 重连策略 | 断线后自动重连，间隔 5 秒，无限重试 |

### 1.2 在线判定

ESP32 **不需要发送心跳包**。服务器通过以下规则判定设备在线/离线：

| 判定规则 | 离线条件 |
|----------|---------|
| 收到来自 ESP32 的任意 JSON 数据包 | 刷新在线状态 |
| 超过 **1 分钟** 未收到任何数据包 | 判定为离线 |

> 因此 ESP32 只要按规定的频率上报状态数据，服务器就能自动知道设备在线，无需额外的心跳机制。

### 1.3 APP 端绑定方式

APP 登录后，用户**手动输入服务器 IP 地址和端口号**（默认 `8.162.21.140:8765`），APP 调用 `POST /api/bind {user_id}` 完成绑定。服务器返回 `{success:true, device_id:"xxx"}`，APP **将 device_id 缓存在内存中**，后续所有操作（查询状态、下发指令）都用这个 `device_id`。

ESP32 端**无需显示二维码**，**无需维护 `device_token`**。服务器绑定成功后，会通过 WebSocket 向 ESP32 发送一条通知：

```json
{"type": "bound", "user_id": "uuid-of-user"}
```

ESP32 收到此消息后可忽略（仅用于 LCD 显示"已绑定"等提示）。

## 2. ESP32 上报数据（ESP32 → 服务器）

### 2.1 上报频率

| 设备状态 | 上报频率 | 说明 |
|----------|---------|------|
| `running`（运行中） | **每 5 秒** | 烹饪进行中，需实时监控温度 |
| `idle`（空闲）/ `stopped`（已停止） | **每 30 秒** | 待机状态，低频上报即可 |

> 此频率已同时满足服务器的 1 分钟在线判定窗口。

### 2.2 运行中状态上报（每 5 秒）

ESP32 在烹饪过程中每 5 秒上报一次当前状态：

```json
{
  "type": "status",
  "temp": 185.5,
  "target_temp": 200,
  "time_left": 415,
  "state": "running",
  "food_name": "鸡翅",
  "fan_speed": "High"
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `type` | string | 固定 `"status"`，用于服务器区分消息类型 |
| `temp` | float | 当前炉内温度（摄氏度），保留 1 位小数 |
| `target_temp` | integer | 目标温度（摄氏度），仅 `state=running` 时发送，idle/stopped 时为 0 |
| `time_left` | integer | 剩余烹饪时间（秒），`state=stopped` 或 `idle` 时为 0 |
| `state` | string | 设备状态：`"running"`（运行中）、`"stopped"`（已停止）、`"idle"`（空闲） |
| `food_name` | string | 当前烹饪食物名称，`state=idle` 时为空字符串 |
| `fan_speed` | string | 风扇转速：`"High"`（高）、`"Medium"`（中）、`"Low"`（低） |

### 2.3 空闲状态上报（每 30 秒）

当空气炸锅待机或烹饪刚停止后：

```json
{
  "type": "status",
  "temp": 26.0,
  "target_temp": 0,
  "time_left": 0,
  "state": "idle",
  "food_name": "",
  "fan_speed": "Low"
}
```

### 2.4 停止后上报

烹饪完成或手动停止后，连续上报 2 次 `state: "stopped"`（间隔 5 秒），确保服务器收到停止通知，之后切换为空闲上报频率（每 30 秒）：

```json
{
  "type": "status",
  "temp": 195.0,
  "target_temp": 200,
  "time_left": 0,
  "state": "stopped",
  "food_name": "鸡翅",
  "fan_speed": "Low"
}
```

## 3. ESP32 接收指令（服务器 → ESP32）

服务器通过 WebSocket 下发指令，ESP32 需监听并解析 JSON 数据。所有指令使用 `type` 字段区分消息类型。

### 3.1 开始烹饪指令

```json
{
  "type": "cook",
  "food": "鸡翅",
  "temp": 200,
  "time": 720,
  "fan_speed": "High"
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `type` | string | 固定 `"cook"` |
| `food` | string | 食物名称 |
| `temp` | integer | 目标温度（摄氏度），范围 40-220 |
| `time` | integer | 烹饪时长（秒），范围 60-3600 |
| `fan_speed` | string | 风扇转速：`"High"` / `"Medium"` / `"Low"` |

> ⚠️ **兼容性说明**：服务器可能下发 `"fan_speed"` 字段。ESP32 代码同时兼容 `"funSpeed"` 旧字段名。

**ESP32 收到后应做**：
1. 保存食物名称和参数
2. 启动加热管，控温到目标温度
3. 启动风扇到指定转速
4. 开始倒计时，并**每 5 秒**上报（state = `"running"`）
5. 倒计时归零后自动停止加热，state 变为 `"stopped"`，连续上报 2 次后切回空闲频率
6. 停止后继续保持 WebSocket 连接，等待下次指令

### 3.2 停止烹饪指令

```json
{
  "type": "stop"
}
```

**ESP32 收到后应做**：
1. 立即停止加热管
2. 风扇继续运行 30 秒散热，然后停止
3. state 变为 `"stopped"`
4. 以 `"stopped"` 状态上报 2 次（间隔 5 秒）
5. 之后切换为空闲上报频率（每 30 秒）

### 3.3 预约烹饪指令

```json
{
  "type": "schedule",
  "food": "牛排",
  "temp": 200,
  "time": 480,
  "fan_speed": "High",
  "scheduled_at": "2026-06-22-11:00:00"
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `type` | string | 固定 `"schedule"` |
| `food` | string | 食物名称 |
| `temp` | integer | 目标温度（摄氏度） |
| `time` | integer | 烹饪时长（秒） |
| `fan_speed` | string | 风扇转速 |
| `scheduled_at` | string | 预约执行时间，格式 `"YYYY-MM-DD-HH:mm:ss"` |

**ESP32 收到后应做**：
1. 保存预约参数到本地
2. 在 `scheduled_at` 指定时间自动开始烹饪
3. 开始烹饪后，按运行状态每 5 秒上报
4. 如果到了预约时间但 ESP32 已在烹饪中，忽略此预约

> 📌 **注意**：服务器会在 `scheduled_at` 时间到达时通过 WebSocket 下发此指令。ESP32 不需要自己维护复杂定时器来触发烹饪，被动接收服务器下发的指令即可。但为防止网络延迟，服务器会在预约时间前 2 秒提前下发，ESP32 收到后等待到精确时间再启动。

### 3.4 绑定通知

```json
{
  "type": "bound",
  "user_id": "uuid-of-user"
}
```

ESP32 收到此消息后可在 LCD 上显示"设备已绑定"等提示，无需回复。

## 4. 完整通信流程示例

### 4.1 APP 绑定与烹饪流程

```
ESP32                         服务器                      APP
  |                              |                          |
  |--- ws连接 ----------------->|                          |
  |--- status (idle, 30s间隔) -->|                          |
  |--- status (idle, 30s间隔) -->|                          |
  |                              |                          |
  |                              |   <-- 用户输入IP:端口绑定 |
  |                              |   <-- 用户设置参数并发送  |
  |                              | <-- POST /api/cook       |
  |                              |                          |
  |   <-- cook指令下发 -----------|                          |
  |                              |                          |
  |--- status (running, 5s) ---->|                          |
  |--- status (running, 5s) ---->|-- GET /api/status/xxx -->| (每2秒轮询)
  |--- status (running, 5s) ---->|                          |
  |        ...                   |         ...              |
  |--- status (stopped, 5s) ---->|                          |
  |--- status (stopped, 5s) ---->|                          |
  |--- status (idle, 30s) ------>|-- 推送通知 ------------->| (FCM/APNs)
```

### 4.2 预约烹饪流程

```
APP                         服务器                     ESP32
 |                             |                          |
 |-- POST /api/schedule ------>|                          |
 |                             |-- 保存预约并启动定时器     |
 |                             |         ...              |
 |                             |-- 定时器触发(提前2秒)     |
 |                             |-- ws下发 schedule指令 --->|
 |                             |                          |
 |                             |   <-- status (running) --| (每5秒)
 |                             |                          |
 |   <-- 每2秒轮询状态 ---------|                          |
```

## 6. 错误处理

| 场景 | ESP32 处理方式 |
|------|---------------|
| WebSocket 断线 | 自动重连，间隔 5 秒递增（最大 60 秒） |
| 烹饪中收到新 cook 指令 | 停止当前烹饪，执行新指令 |
| 温度传感器故障 | state 变为 `"stopped"`，上报停止状态 |
| 温度超过 250°C（过热保护） | 立即停止加热，state 变为 `"stopped"` |
| 网络长时间断线（>5分钟） | 按本地参数继续完成烹饪（不依赖服务器） |

## 7. ESP32 启动完整流程

```
1. 上电
2. 初始化 LCD、温度传感器、WiFi
3. 如果是首次启动（无 WiFi 凭据）：
   a. LCD 显示 WiFi 配网二维码（用户扫码输入 WiFi 密码）
   b. 配网成功，保存凭据到 NVS
4. 连接 WiFi
5. LCD 显示「已联网」及本机 IP 地址
6. 建立 WebSocket 连接到 ws://8.162.21.140:8765
7. 发送首次状态上报（state: "idle"）
8. 进入主循环：
   - 监听 WebSocket 指令（cook / stop / schedule / bound）
   - 运行中每 5 秒上报状态
   - 空闲时每 30 秒上报状态
   - 更新 LCD 显示（温度、时间、状态、风扇转速）
   - 无需发送心跳包（状态上报即心跳）
```
